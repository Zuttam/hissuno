# Brief Interpretation

The `brief` field on the issue is the primary implementation spec. It's generated by Hissuno's brief workflow when the issue transitions to `ready`.

## What the brief contains

A brief typically covers:
- **Problem statement** - what's broken or missing, grounded in customer signals
- **Proposed change** - what to build / fix at a product-behavior level
- **Files likely affected** - hints; not exhaustive or binding
- **Acceptance criteria** - observable behavior that defines "done"
- **Out of scope** - explicit boundaries; respect these

Briefs vary in depth. Some are terse lists, some are multi-section docs.

## How to read it

1. Find the acceptance criteria first. If absent, derive them from the problem statement and confirm with the user before coding.
2. Take the file hints as a starting point, not a hard list. Grep the codebase for the behavior described and let the real call sites guide you.
3. Respect "out of scope" lines strictly. Those are almost always there because a previous attempt over-reached.
4. If the brief references other issues, feedback, or scopes, pull them up:
   ```bash
   hissuno get issues <related-id>
   hissuno get feedback <session-id>
   hissuno get scopes <scope-id>
   ```

## Scope discipline

After implementation, re-read the brief and walk the diff. Every changed file must trace back to a sentence in the brief (or to an obvious consequence of one — e.g. a test for a function mentioned in the brief). Delete anything that doesn't.

## When the brief is unclear

Stop and ask the user. Do **not** guess. The cost of a short clarification round beats a PR that ships the wrong feature and has to be reverted.

If the user isn't available, leave the issue in `in_progress` and report what needs clarification. The next iteration can resume or a human can edit the brief and re-run.

## When the brief is missing

`brief` is null even though status is `ready`. This shouldn't happen under normal conditions — if you see it, surface to the user rather than proceeding.
