## Overview

Knowledge packages are the compiled output of the analysis workflow. They take the raw information extracted from your knowledge sources and organize it into structured categories that the Hissuno Agent can reference when answering customer questions.

Each package is a Markdown document stored in secure cloud storage, versioned automatically, and linked to your project. When a customer asks a question, the Hissuno Agent retrieves the relevant package content to formulate an accurate, context-aware response.

## Package Categories

Every analysis run produces five distinct knowledge packages, each serving a different purpose:

### Business

Contains company-level information such as pricing details, policies, contact information, company background, and go-to-market positioning. The Hissuno Agent uses this package to answer questions about plans, billing, partnerships, and organizational matters.

### Product

Covers features, capabilities, use cases, and limitations of your product. This is typically the most referenced package during customer conversations. It helps the agent explain what the product does, how features work, and what is or is not possible.

### Technical

Includes API references, architecture details, data models, integration guides, and infrastructure information. This package enables the agent to assist with technical questions, troubleshoot integration issues, and explain how systems connect.

### FAQ

A structured collection of frequently asked questions and their answers, organized by topic. The FAQ package is generated by identifying common question patterns across your source materials and synthesizing clear, direct answers.

### How-To

Step-by-step tutorials, getting-started guides, and best practice documentation. The agent uses this package to walk customers through specific procedures, from initial setup to advanced configuration.

## How Packages Power the Hissuno Agent

When the Hissuno Agent receives a customer message, it follows this process:

1. **Semantic search**: The message is converted to a vector embedding and compared against indexed knowledge chunks to find the most relevant content
2. **Package retrieval**: The matched content spans one or more package categories, giving the agent context from multiple angles (product, technical, FAQ, etc.)
3. **Response generation**: The agent synthesizes a response using the retrieved knowledge, ensuring answers are grounded in your actual product documentation and codebase

This approach means the agent does not hallucinate or guess. Its responses are anchored to the specific knowledge you have provided through your sources.

## Viewing Packages

You can view the contents of each knowledge package from the **Agents** page:

1. Navigate to the **Agents** page from the sidebar
2. Click on a knowledge package to open the Knowledge Detail dialog
3. Each package category is listed with its version number and generation timestamp
4. Click on a category to view the full Markdown content

The content viewer shows the compiled knowledge exactly as the Hissuno Agent sees it. This lets you verify accuracy and identify gaps before the agent uses the information in live conversations.

## Package Versioning

Each package category is versioned independently. Every time you run an analysis, the version number increments for all categories. Previous versions are replaced in storage; only the latest version is retained.

The version number and generation timestamp are tracked in the database, so you can see when the package was last updated and how many analysis runs have been performed.

## Named Packages

For projects that need multiple distinct knowledge sets, you can create named packages. A named package groups specific knowledge sources together and produces its own set of five category packages.

### When to Use Named Packages

- **Multi-product projects**: If your project covers more than one product, each product can have its own named package with dedicated sources
- **Audience segmentation**: Create separate packages for different user types (e.g., developers vs. business users)
- **Regional variations**: Maintain separate packages for different regions or markets

### Managing Named Packages

1. Navigate to the **Agents** page from the sidebar
2. Create a named package with a descriptive name and optional description
3. Link knowledge sources to the package
4. Optionally add **guidelines** that influence how the compiler categorizes information
5. Run analysis to generate the package

Each named package tracks its own source count, last analysis timestamp, and generated categories.

## Vector Embeddings

After packages are compiled and saved, the workflow generates vector embeddings for semantic search. The content is split into chunks, and each chunk is embedded so the Hissuno Agent can perform similarity searches against customer questions.

This indexing step runs automatically as the final phase of the analysis workflow. You do not need to configure or manage embeddings manually.

## Security and Sanitization

Before packages are saved, they pass through a security sanitization step. The Security Scanner agent scans all compiled content for sensitive information and redacts it automatically:

- AWS keys and access tokens
- API keys and secrets
- Database connection strings
- Passwords and credentials
- Private keys and internal IP addresses
- GitHub tokens and other service credentials

Redacted values are replaced with descriptive placeholders like `[REDACTED_API_KEY]` or `[REDACTED_DATABASE_URL]`. A summary of redactions is logged, including the total count and the types of sensitive data found across each category.

This ensures that secrets from your codebase or documentation never leak into the knowledge that the Hissuno Agent shares with customers.

## Refreshing Packages

Knowledge packages reflect the state of your sources at the time of analysis. To keep the agent current:

- Re-run analysis after shipping major features or updating documentation
- Monitor the generation timestamp to know how recent the agent's knowledge is
- Review package contents periodically to catch outdated or inaccurate information

See the [Analysis Workflow](/docs/knowledge/analysis) documentation for details on triggering re-analysis.
