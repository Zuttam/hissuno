# Marketing Plan: “Support that ships” (0 → 1,000 customers at $50/mo)

Goal (Milestone #1): 1,000 paying customers × $50/mo = $50,000 MRR
Primary buyer: founders and small teams who feel support interrupts engineering
Core promise: answer support with code context, then convert real issues into shipped fixes

⸻

1) Positioning

Positioning line: Close the loop from customer support to shipped code.

One-sentence pitch:
Support that understands your codebase and turns high-value tickets into specs, issues, and reviewable PRs.

Differentiator:
	•	Generate support knowledge directly from the repo and architecture
	•	Ground answers in implementation details (not generic docs)
	•	Escalate “worth fixing” tickets into structured engineering artifacts (spec → issue → PR)

⸻

2) ICP and targeting

ICP v1 (best fit for $50/mo)
	•	Team size: 2–15 people (often 1–5 engineers)
	•	Business model: B2B SaaS, API products, integrations-heavy tools, technical SMB software
	•	Support profile: 10–500 conversations/month, many “how does X work” and “why is this failing” tickets
	•	Pain: founders/engineers get pulled into support, slow turnaround, poor documentation

Avoid early
	•	Consumer apps (billing/account heavy, less code-context need)
	•	Very early indie projects with near-zero support volume
	•	Enterprise compliance-heavy environments (procurement, security reviews)

3 micro-niches to start (pick one)
	1.	API + SDK products (debugging, auth, webhooks, integrations)
	2.	B2B SaaS with integrations (CRM, payments, SSO, Slack/Zapier)
	3.	Devtools-lite SaaS (analytics, monitoring, dashboards for SMBs)

4) Packaging for $50/mo and self-serve

Pricing (simple)

	•	Code-grounded answers + issue drafts
	•	Pro — $99/mo
	•	Multiple repos/inboxes
	•	Higher cap (example: 1,000/mo)
	•	PR creation, Slack alerts, custom rules
	•	Team — $199+/mo
	•	SSO, roles, audit logs, more advanced workflows

Trial
	•	Usage-based trial: “first 50 conversations free” or “7 days + 50 conversations”
	•	In-product progress: “X answers grounded in repo”, “Y escalations drafted”

Keep your own support load low
	•	Onboarding checklist (repo → inbox → first grounded answer → first issue draft)
	•	Safe defaults: cite code sources, admit uncertainty, escalate instead of guessing

5) Distribution strategy (to reach 1,000)

You won’t reach 1,000 from founder communities alone.
Plan: PLG + integration directories + SEO + templates, with targeted outbound to seed.

Channel 1: Integration directories (high intent)
	•	GitHub Marketplace
	•	Intercom App Store
	•	Zendesk Marketplace
	•	Plus “works with email” as fallback

Channel 2: SEO (compounds)

Target intent queries:
	•	“answer Intercom with code context”
	•	“reduce engineering escalations support”
	•	“turn support tickets into GitHub issues”
	•	“customer support to PR workflow”

Content that converts:
	•	Integration pages (rank + convert)
	•	Playbooks (“Stop forwarding support to engineering”)
	•	Templates (“Ticket → Issue with repro + file pointers”)
	•	Case studies (small teams shipping faster)

Channel 3: Community seeding (fast early, not scalable)
	•	Indie Hackers, HN Show, r/SaaS, founder Slack/Discords
	•	Goal: first 20–50 design partners + testimonials + screenshots

Channel 4: Targeted outbound (surgical)
	•	Reach out to B2B SaaS with technical support presence
	•	Pitch: “reduce escalations, try it in 10 minutes”
	•	Keep volume low: 10–30 calls/month max

6) Launch plan and milestones

Phase 0: Pre-launch (0 → 20 paying)

Objective: validate ICP + messaging + onboarding
Deliverables:
	•	Landing page + 3-min demo
	•	Connect GitHub + connect inbox + first grounded answer
	•	“Escalate” creates issue draft with repro + context

Success metrics:
	•	40%+ of trials reach “first grounded answer” within 24 hours
	•	20%+ trial-to-paid for best-fit ICP

Phase 1: Early growth (20 → 200 paying)

Objective: prove retention + build repeatable loop
Deliverables:
	•	Directory listings + early reviews
	•	5–10 SEO pages that convert
	•	3–5 case studies with numbers (time saved, fewer escalations)

Success metrics:
	•	60-day gross revenue retention ≥ 85%
	•	Weekly active usage per account (inbox connected + answers generated)

Phase 2: Scale (200 → 1,000 paying)

Objective: add leverage, not complexity
Deliverables:
	•	More integrations (Zendesk, Intercom, email first-class)
	•	Improve “spec → issue → PR” reliability
	•	Partnerships (agencies, support consultancies, dev communities)

Success metrics:
	•	Net revenue retention trends > 100% via upgrades
	•	Organic signups grow MoM without proportional manual effort

7) Funnel design (click → paid)

TOFU
	•	Integration pages, templates, short demos, screenshots
	•	Primary CTA: Connect your repo

Activation (AHA moment)

AHA definition: at least one answer grounded in repo AND one escalation artifact generated
Steps:
	1.	Connect repo
	2.	Connect inbox
	3.	Ask a real support question
	4.	Click “Create issue draft” on a real ticket

Conversion
	•	Paywall after trial usage threshold
	•	Show value counters: answers resolved, escalations drafted, interruptions prevented

Retention
	•	Weekly digest: top issues, top unanswered questions, suggested fixes
	•	Consistent “engineering-ready” issue format
	•	Avoid overconfident wrong answers

13) Risks and mitigations

Risk: low support volume means low urgency

Mitigation: sell “protect maker time” and “reduce interrupts,” not “handle volume.”

Risk: wrong answers kill trust

Mitigation: cite repo sources, show uncertainty, escalate by default when unsure.

Risk: PR generation isn’t reliable early

Mitigation: win on issue drafts + file pointers + suggested patch view, add PR creation when stable.

Risk: indie audience is price sensitive

Mitigation: keep Starter affordable, but make Pro clearly valuable for real support load.
