### Comment 1: On Gergely Orosz - Agent-first dev platform

**Post context**: GitHub's last CEO building EntireHQ, agent-first dev platform, $60M seed

```
The "agent-first" framing is interesting because it inverts the usual pitch.

Most AI dev tools start with "make the developer faster." This seems to start with "redesign the workflow around agent capabilities."

The gap I keep seeing in practice: agents are exceptional at generating code but still blind to the context that makes code useful - the customer workaround that drove the requirement, the production edge case, the decision history behind the architecture.

Curious whether EntireHQ's "Checkpoints" approach captures that context or focuses on the code layer. Because the teams I've seen move fastest aren't the ones with better agent tooling. They're the ones where the agent has access to the full picture of why something needs to be built.
```

### Comment 2: On Lenny Rachitsky - Meta's "Product Sense with AI" interview

**Post context**: Meta added AI-focused PM interview, "Product Sense with AI"

```
This is the right signal. Product sense about AI is fundamentally different from product sense with AI tools.

The hard part isn't learning to use AI in your workflow. The hard part is developing intuition for what AI can and can't solve reliably - which customer problems benefit from AI, where deterministic logic is still better, and when the 90% accuracy case creates more support tickets than it prevents.

From the engineering side, the PMs who've been most effective working with AI systems are the ones who understand failure modes, not just capabilities. They ask "what happens when the model gets it wrong" before "what happens when it gets it right."

Meta testing for this explicitly tells me we're past the "AI as feature" phase and into the "AI as product architecture" phase. That's a meaningful shift.
```

### Comment 3: On John Cutler - Internal platforms

**Post context**: Progression from untreated internal platform to productized platform

```
The jump from 2 to 3 is where most teams get stuck. And it's almost always for the same reason.

An internal platform treated like a product still has a captive audience. Your internal users can't churn. They can complain, but they can't leave. So the feedback signal is fundamentally different from an external product.

The platforms that make it to stage 3 are usually the ones that built a real feedback loop early - not a survey, not a Slack channel, but actual instrumentation of how people use (and work around) the platform.

The pattern I've seen: the best internal platforms were built by teams who treated their colleagues like customers whose feedback decays at every handoff. Same problem, different audience.
```

### Comment 4: On Lenny Rachitsky - "Should I start learning to code"

**Post context**: Lenny sharing a take that learning to code is optimizing for the wrong skillset

```
As someone who writes code every day - there's nuance here worth unpacking.

Learning to code isn't about the syntax anymore. AI handles that. But understanding systems, constraints, and tradeoffs? That's more valuable than ever.

The people I see thriving aren't the ones who can write code from scratch. They're the ones who can read code, reason about architecture, and define constraints for AI to work within. That's a different skill than "learning to code" in the traditional sense.

The real skill gap isn't code vs. no-code. It's understanding the shape of a technical problem well enough to know when the AI-generated solution will break in production.
```
